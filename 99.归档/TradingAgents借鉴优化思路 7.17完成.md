# TradingAgents 借鉴思路总结

**来源：** https://github.com/TauricResearch/TradingAgents

**时间：** 2026-07-14

**目标：** 借鉴该多智能体交易框架的设计，优化 BTC/ETH 日内合约分析 SOP

---

## 1. 多空辩论机制

**核心想法：** 让多头和空头作为两个独立的 LLM 智能体互相对话、逐条反驳，最后由一个裁判综合出结论。对比目前一个人在 prompt 里"自己和自己说两套"的思路，独立辩论的对抗性更强，论据质量更高。

**技术可行性：** 同一个 API 完全可以驱动。一次辩论约 3-4 次 API 调用：

1. 多头出论点
2. 空头出论点
3. 多头反驳空头 / 空头反驳多头
4. 裁判综合

如果用快速模型，总耗时约 10-20 秒，在日内分析的接受范围内。

**与 SOP v3.3 的关联：** v3.3 原则 9 已要求输出两套路径，辩论机制是这条原则的自然升级——从"一个人写两方面"变成"两个角色互辩"。

---

## 2. 结构化输出（Pydantic Schema）

**核心想法：** 把 SOP 里的 markdown 输出模板（第 5、7、10 节）转成类型化的数据模型。每个字段有类型约束、字段描述充当 LLM 的输出指令，确保跨运行、跨模型的一致性。

**技术可行性：**

- 改动量很小——现有 markdown 模板的字段直接映射成 Pydantic field
- 内部用 schema 传数据，最终展示时仍可反渲染回 markdown 给人类读
- AI 读结构化数据比正则从自由文本里提取更可靠，下游做统计、记录、回测都更自然

**与 SOP v3.3 的关联：** 模板不变，只是内部格式升级。第 10 节交易计划、第 5 节技术分析、第 7 节支撑阻力都可以优先试用。

---

## 3. 交易日志与复盘机制

**核心想法：** 让 SOP 从"每次独立分析"变为"持续学习"。每笔交易的结果和复盘记录被持久化，下次分析时注入历史教训作为上下文。

**实施路径（建议从方案 A 起步）：**

**方案 A（推荐起步）：** 本地 markdown 日志文件

- 格式简单，每次分析追加一条记录
- 下次运行同类标的时注入最近的 3-5 条作为上下文
- 维护成本几乎为零

**方案 B（中期升级）：** JSONL 结构化日志

- 每笔交易一条 JSON 行，字段固定
- 可自动计算胜率、R:R 分布、方向胜率等统计

**方案 C（远期）：** 数据库（SQLite）

- 支持回测分析和系统性策略优化
- 适合已有持续自动化管线后考虑

**与 SOP v3.3 的关联：** v3.3 有连续亏损停手规则（第 9.5 节），但缺少学习机制。日志复盘是让这套规则"长出记忆"。

---

## 4. 双层 LLM 策略

**核心想法：** 不同角色用不同能力的模型。分析师和辩论员用快速便宜模型，裁判和最终决策用深度推理模型。

**技术可行性：** 同一个 API 提供商，切换 model 参数即可。不需要两套 API key。

| 角色 | 推荐模型 | 原因 |
|---|---|---|
| 市场/数据分析节点 | 快速模型 | 调用量大，不需要深度推理 |
| 多空辩论员 | 快速模型 | 同上，辩论本身靠 prompt 质量 |
| 方向综合判断（裁判） | 深度推理模型 | 需要权衡矛盾论点 |
| 最终交易计划拍板 | 深度推理模型 | 需要全面评估风险 |

**与 SOP v3.3 的关联：** 如果后续把 SOP 自动化成一条管线，"分析节点用低配模型、决策节点用高配模型"是合理的成本策略。目前手动分析阶段天然已存在"强 LLM 做判断 + 工具做数据"的分层。

---

## 几个可能的 v3.4 方向

按实施难度排序，可以做参考：

1. **交易日志**——先写一个简版复盘记录文件，代价最小，长期收益最大
2. **结构化输出**——选一个模板（如第 10 节交易计划）试点转成 schema
3. **多空辩论**——在现有 prompt 基础上，增加独立的多空辩论环节而不是一个人两面写
4. **双层 LLM**——如果后续自动化再考虑，手动分析阶段还不是硬需求
